Configure the Nuxt Convex module in your `nuxt.config.ts`.

## Module Options

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['nuxt-convex'],
  convex: {
    dir: 'convex',
    url: 'https://your-deployment.convex.cloud',
    server: true,
    storage: true,
    r2: true,
  },
})
```

| Option    | Type      | Default                  | Description                                                              |
| --------- | --------- | ------------------------ | ------------------------------------------------------------------------ |
| `dir`     | `string`  | `'convex'`               | Convex source directory. Relative paths are resolved across Nuxt layers. |
| `url`     | `string`  | `process.env.CONVEX_URL` | Convex deployment URL                                                    |
| `server`  | `boolean` | `true`                   | Enable SSR queries by default                                            |
| `storage` | `boolean` | `false`                  | Enable file storage integration                                          |
| `r2`      | `boolean` | `false`                  | Enable Cloudflare R2 integration                                         |

## Layer Resolution

When `convex.dir` is relative, `nuxt-convex` checks the app layer first and then each extended layer in Nuxt priority order.

- Reads prefer the first layer that already contains generated Convex API files.
- If no generated layer exists yet, the module falls back to the first matching directory and warns for that exact location.
- Writes and scaffolding stay in the resolved directory when it is local and writable.
- If the resolved directory is read-only or comes from an installed dependency layer, scaffolding falls back to the project layer and keeps imports pointed at the authoritative generated layer.

Set `convex.dir` explicitly when you want a custom location such as `backend/convex`.

## Environment Variables

| Variable                 | Description                                     |
| ------------------------ | ----------------------------------------------- |
| `CONVEX_URL`             | Convex deployment URL (set by `npx convex dev`) |
| `NUXT_PUBLIC_CONVEX_URL` | Alternative, follows Nuxt convention            |

The module reads `CONVEX_URL` first, then falls back to `NUXT_PUBLIC_CONVEX_URL`. Running `npx convex dev` sets these automatically.

## Virtual Modules

The module provides these virtual imports:

| Import            | Purpose                           |
| ----------------- | --------------------------------- |
| `#convex`         | Core composables and client       |
| `#convex/api`     | Generated Convex API types        |
| `#convex/storage` | Storage operations (when enabled) |
| `#convex/r2`      | R2 Vue helpers (when enabled)     |

```ts
import { useConvexMutation, useConvexQuery } from '#convex'
import { api } from '#convex/api'
```

## Auto-Imports

These composables are auto-imported globally:

| Composable          | Description                |
| ------------------- | -------------------------- |
| `useConvexQuery`    | Subscribe to reactive data |
| `useConvexMutation` | Execute write operations   |
| `useConvexAction`   | Call async actions         |
| `useConvexClient`   | Get raw client             |

When storage is enabled:

| Composable         | Description                  |
| ------------------ | ---------------------------- |
| `useConvexStorage` | Low-level storage operations |
| `useConvexUpload`  | File upload with progress    |

When R2 is enabled:

| Composable          | Description                   |
| ------------------- | ----------------------------- |
| `useConvexR2Upload` | Upload files to Cloudflare R2 |

## DevTools

In development, the module adds a **Convex** tab to Nuxt DevTools with an embedded Convex dashboard. Access your data, run functions, and view logs directly from DevTools.

## TypeScript

The module generates types automatically. If you need to reference types in your code:

```ts
import type { Doc, Id } from '#convex/api'

// Document type
type Task = Doc<'tasks'>

// ID type
type TaskId = Id<'tasks'>
```
