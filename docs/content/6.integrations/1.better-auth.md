Use this integration when your Nuxt app needs Better Auth on the server and Convex as the database. The current supported path is email/password. The Nuxt playground landing page and the `Session & Diagnostics` dashboard section validate that path today.

::important
This integration runs on the Nuxt server. It is not available in a client-only setup.
::

## Current support status

This integration is currently dependency-constrained. The provider hook and HTTP adapter ship in this repo, but the end-to-end stack still depends on external packages that the workspace tracks through its current catalog pins.

Treat this page as documentation for the shipped wiring, not as a stable release matrix. If you need a known-good combination today, match the package versions used by the playground in this repository.

The supported baseline is email/password. GitHub auth can remain available behind environment flags, but it is optional and secondary. Do not treat social auth as part of the canonical integration contract yet.

## Install the tested Better Auth stack

::warning
The tested stack is published on npm, but it is still not a fully stable release matrix. `@onmax/nuxt-better-auth` remains an alpha release, so keep these versions pinned together and update them as a set when the playground baseline changes.
::

```bash
pnpm add @onmax/nuxt-better-auth@0.0.2-alpha.31 \
  @convex-dev/better-auth@0.11.3 \
  better-auth@1.5.6
```

This command matches the versions the playground currently exercises. Update the pins together when the tested integration stack changes.

## Understand what each package owns

- `nuxt-convex` registers the `convex` database provider on the `better-auth:database:providers` hook.
- `nuxt-convex` also exports `createConvexHttpAdapter` from `nuxt-convex/better-auth`.
- `@onmax/nuxt-better-auth` owns the Nuxt-side auth routes, session client, and route middleware.
- `@convex-dev/better-auth` owns the Convex-side adapter and auth component wiring.

## Configure Nuxt

Configure Nuxt so Better Auth asks for the `convex` provider. `nuxt-convex` then exposes the resolved Convex URL through runtime config, and `auth.database.convexUrl` remains the higher-priority override when you need it.

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['nuxt-convex', '@onmax/nuxt-better-auth'],
  auth: {
    database: {
      provider: 'convex',
    },
  },
})
```

## Add environment variables

Add the minimum environment for the supported email/password flow first.

```bash [.env]
CONVEX_URL=https://your-project.convex.cloud
BETTER_AUTH_SECRET=replace-with-a-secret-at-least-32-characters
```

If you later enable GitHub as an optional secondary provider, add the GitHub client ID and secret separately. Keep those variables out of the baseline setup until you actually need that branch.

## Enable email/password in Convex

The supported auth path depends on the Convex auth component enabling email/password explicitly inside its existing `betterAuth(...)` call.

```ts [convex/auth.ts]
emailAndPassword: {
  enabled: true,
  requireEmailVerification: false,
},
```

## Create auth config files

Create the Nuxt-side auth config files after the provider and environment are in place.

```ts [server/auth.config.ts]
import { defineServerAuth } from '@onmax/nuxt-better-auth/config'

export default defineServerAuth(() => ({
  appName: 'nuxt-convex app',
}))
```

```ts [app/auth.config.ts]
import { convexClient } from '@convex-dev/better-auth/client/plugins'
import { defineClientAuth } from '@onmax/nuxt-better-auth/config'

export default defineClientAuth(({ siteUrl }) => ({
  baseURL: new URL('/api/auth', siteUrl).toString(),
  plugins: [convexClient()],
}))
```

## What ships in this repo

`nuxt-convex` exports `createConvexHttpAdapter` from `nuxt-convex/better-auth`. The adapter validates the Convex URL, maps Better Auth record shapes, and paginates reads through the Convex HTTP client.

::note
Use the playground as the validation reference when you wire this up. The sign-in page exercises the email/password entry flow, and the dashboard confirms the current session and runtime state after login.
::
