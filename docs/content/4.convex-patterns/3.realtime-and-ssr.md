---
title: Realtime and SSR
description: Understand how Convex queries render on the server, hydrate on the client, and stay subscribed after the initial response.
---

`useConvexQuery` always works in two phases. It can fetch once through the HTTP client during SSR, then it subscribes through the browser client on the client side.

## Server phase

When `server` is enabled, the composable uses `ConvexHttpClient` during SSR.

```ts
const { data, suspense } = useConvexQuery(api.tasks.list, { userId: 'demo' })
await suspense()
```

The server result populates the initial HTML and payload.

## Client phase

After hydration, the composable seeds from the local cache when possible and subscribes with `client.onUpdate(...)`.

That subscription keeps `data` fresh without polling.

## When to disable SSR

Use `{ server: false }` or set `convex.server = false` when:

- the query needs browser-only auth or runtime state
- the page is private and SEO does not matter
- you want to avoid server-side Convex requests entirely

::important
Authenticated data is usually a client-only concern with the current shared helpers because server-side query execution does not automatically forward browser auth state.
::

## Paginated queries

`useConvexPaginatedQuery` is client-only. It subscribes through realtime pagination and does not perform the same SSR prefetch step as `useConvexQuery`.
