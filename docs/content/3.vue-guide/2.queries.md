---
title: Queries
description: Subscribe to Convex queries with reactive arguments, SSR prefetching, and automatic realtime updates.
---

`useConvexQuery` subscribes to a Convex query and keeps the result in sync whenever the backend data changes. On the server, it fetches once through the HTTP client. On the client, it opens a realtime subscription.

## Basic usage

```vue [src/components/TaskList.vue]
<script setup lang="ts">
import { useConvexQuery } from '@onmax/convex-vue'
import { api } from '../convex/_generated/api'

const { data, error, isPending, suspense } = useConvexQuery(
  api.tasks.list,
  { userId: 'user_123' },
)

await suspense()
</script>
```

## Return values

::field-group
  ::field{name="data" type="Readonly<Ref<T | undefined>>"}
  The latest query result. Starts as `undefined` until the first response arrives.
  ::

  ::field{name="error" type="Readonly<Ref<Error | null>>"}
  The latest error from the query subscription. Resets to `null` on the next successful result.
  ::

  ::field{name="isPending" type="ComputedRef<boolean>"}
  `true` while `data` is `undefined` and no error has occurred. Always `false` when skipped.
  ::

  ::field{name="isSkipped" type="ComputedRef<boolean>"}
  `true` when the query is disabled via `"skip"`.
  ::

  ::field{name="suspense" type="() => Promise<T | undefined>"}
  Resolves when data is available. On the server, it triggers an HTTP fetch. On the client, it waits for the first subscription result. Resolves `undefined` immediately when skipped.
  ::
::

## Reactive arguments

Pass a `computed`, a `ref`, or a getter function as the second argument. The composable re-subscribes whenever the arguments change.

```vue [src/components/UserTasks.vue]
<script setup lang="ts">
import { useConvexQuery } from '@onmax/convex-vue'
import { api } from '../convex/_generated/api'

const props = defineProps<{ userId: string }>()

const { data } = useConvexQuery(
  api.tasks.list,
  computed(() => ({ userId: props.userId })),
)
</script>
```

The previous subscription is cleaned up automatically before the new one starts.

## SSR and `suspense()`

When `server` is `true` (the default), `suspense()` fetches data through `ConvexHttpClient` during server rendering. The client receives the prefetched result and opens a realtime subscription on top of it.

Call `await suspense()` in your `<script setup>` to block rendering until the data is ready. Skip the call when you prefer to show a loading state immediately.

## Client-only mode

Disable SSR for a single query by passing `{ server: false }`.

```ts
const { data } = useConvexQuery(api.tasks.list, { userId: 'demo' }, { server: false })
```

The composable skips the HTTP fetch and waits for the realtime subscription to deliver the first result. Use this for authenticated queries where the server has no session context.

## Error handling

```vue [src/components/TaskList.vue]
<script setup lang="ts">
import { useConvexQuery } from '@onmax/convex-vue'
import { api } from '../convex/_generated/api'

const { data, error, isPending } = useConvexQuery(api.tasks.list, {})
</script>

<template>
  <p v-if="isPending">Loading...</p>
  <p v-else-if="error">{{ error.message }}</p>
  <ul v-else>
    <li v-for="task in data" :key="task._id">{{ task.text }}</li>
  </ul>
</template>
```

The `error` ref updates whenever the subscription reports a failure and resets on the next successful result.

## Conditional queries with `"skip"`

Pass `"skip"` as the arguments to disable the query without removing it from your component. The composable unsubscribes, sets `data` to `undefined`, and `isSkipped` becomes `true`.

```vue [src/components/UserProfile.vue]
<script setup lang="ts">
import { useConvexQuery } from '@onmax/convex-vue'
import { api } from '../convex/_generated/api'

const props = defineProps<{ userId?: string }>()

const { data, isSkipped } = useConvexQuery(
  api.users.get,
  computed(() => props.userId ? { userId: props.userId } : 'skip'),
)
</script>
```

When `userId` becomes available, the composable subscribes automatically. Use this pattern for queries that depend on async data like authentication state.
