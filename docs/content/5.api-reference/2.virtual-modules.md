---
title: Virtual Modules
description: Understand the Nuxt-only aliases created by nuxt-convex and when they become available.
---

`nuxt-convex` creates aliases so Nuxt app code does not have to reach into generated or build directories directly. These aliases define the public import paths for the Nuxt wrapper.

## Always available

These aliases are always available when the module is installed.

| Import        | What it exposes                                                       |
| ------------- | --------------------------------------------------------------------- |
| `#convex`     | Re-exports from `@onmax/convex-vue` through the Nuxt build directory. |
| `#convex/advanced` | Re-exports from `@onmax/convex-vue/advanced`.                   |
| `#convex/api` | Generated Convex API types from your resolved `convex.dir`.           |

```ts [app/pages/tasks.vue]
import { api } from '#convex/api'

const { data } = useConvexQuery(api.tasks.list, { userId: 'demo' })
```

## Available when `storage: true`

Enable `storage: true` to add the storage alias.

| Import            | What it exposes                              |
| ----------------- | -------------------------------------------- |
| `#convex/storage` | Re-exports from `@onmax/convex-vue/storage`. |

`#convex/storage-refs` stays internal to the Nuxt runtime. Application code should import from `#convex/storage` instead.

## Available when `r2: true`

R2 does not add any virtual modules. Enable `r2: true` to auto-import `useConvexR2Upload` instead.

::note
Use the shared composables from `#convex` inside Nuxt. Direct `@onmax/convex-vue` imports are for standalone Vue apps, not Nuxt app code.
::
